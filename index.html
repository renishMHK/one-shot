How It Works

1

Ticket Enters

A JIRA ticket labeled PFOS is detected by the Specter orchestrator during its polling cycle and routed into the OneShot pipeline.

2

Qualification Gate

The QUALIFIER agent evaluates scope, feasibility, and repository context. Ambiguous or unbounded tickets are rejected with detailed feedback posted back to JIRA.

3

Autonomous Pipeline

Plan arrow_forward Review arrow_forward Implement arrow_forward Code Review arrow_forward Test

Each stage is powered by a specialized AI agent. Review loops are bounded to a maximum of 3 iterations to prevent infinite correction cycles and ensure predictable completion.

4

Human Review

A draft pull request is delivered to GitHub. The pipeline summary, PR link, and completion status are posted back to the original JIRA ticket. A human reviewer takes it from here — OneShot never merges code automatically.

The Agent Roster

Six specialized agents, two model families. Using both Claude and GPT-5.4 provides vendor diversity — reducing the risk of shared blind spots between the agent that writes code and the one that reviews it.

QUALIFIER

Claude

Evaluates ticket scope and feasibility before pipeline entry. Inspects the repository to verify implementation details are discoverable.

PLANNER

Claude

Generates step-by-step implementation plans from ticket context. Maps acceptance criteria to implementation steps with TDD awareness.

REVIEWER

GPT-5.4

Cross-model review for plan quality, correctness, and completeness. Issues verdicts: Approved, Needs Changes, or Blocked.

IMPLEMENTER

Claude

Writes production code on dedicated branches following approved plans. Handles git operations, commits, and branch management.

CODE REVIEWER

GPT-5.4

Reviews generated code for bugs, standards violations, security issues, and regression risks. Can flag issues for revision.

TESTER

Deterministic

Executes automated test suites deterministically (no LLM). Parses test reports, captures coverage metrics, and creates the draft PR.

Preview Deployments

Once a draft PR is created, stakeholders can preview the changes in a live environment before merging — no local setup required.

rocket_launch

How Preview Deploy Works

1
PR Completes
OneShot finishes the pipeline and opens a draft PR on GitHub. The run enters the pr_open state.
2
One-Click Deploy
From the OneShot dashboard, click the deploy button on any eligible run. The request triggers a GitHub Actions workflow to build and push the container image.
3
ECS Deployment
The built image is deployed to a dedicated ECS preview slot. The preview URL becomes available once the service is healthy.
4
Stakeholder Review
Stakeholders access the preview URL to validate changes in a real environment. No git knowledge or local setup needed.

Deployment Status Tracking

Every deployment is tracked with a traffic-light indicator on the dashboard, so teams always know the current state at a glance.

Green — Preview is live and accessible
Amber — Build or deployment in progress
Red — Deployment failed
Grey — Not yet deployed

Slot Management

Preview environments use a managed slot system to control infrastructure costs and avoid conflicts:

  • check_circle One active preview per slot — deploying a new run automatically supersedes the previous one
  • check_circle Smooth handoff — old preview stays live until the replacement is fully healthy
  • check_circle Only one in-flight deployment per slot — prevents conflicting simultaneous builds
